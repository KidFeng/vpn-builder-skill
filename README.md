# vpn-builder-skill — A Claude Code Skill

> Deploy a self-hosted, GFW-resistant VPN with sing-box. Battle-tested playbook for **VLESS + Reality** (TCP/443) + **Hysteria2 + salamander obfs** (UDP/443) dual-stack, with smart Chinese / international traffic splitting on macOS / iOS / Android / Linux.

This is a [Claude Code](https://claude.com/claude-code) **skill** — a structured playbook + templates + reference docs. After installing it, you ask Claude "搭建 VPN" (or "build me a VPN"), and Claude walks you through a 6-stage workflow that ends with a working VPN.

---

## Quick start (happy path)

You'll need a Mac/Linux workstation with Claude Code, an SSH-accessible Linux VPS outside mainland China, and ~1 hour for the first deploy (mostly review time).

```bash
# 1. Install the skill (one-time)
mkdir -p ~/.claude/skills
git clone https://github.com/KidFeng/vpn-builder-skill.git ~/.claude/skills/vpn-builder-skill

# 2. (optional but recommended) point a domain at your server
#    e.g., add an A record `tokyo.yourdomain.com → 203.0.113.10`

# 3. Open Claude Code and say:
"build me a VPN on tokyo.yourdomain.com (IP 203.0.113.10)"
# or in Chinese: "在 tokyo.yourdomain.com (203.0.113.10) 上搭建VPN"
```

Claude will trigger the `vpn-builder-skill` and walk through:

1. **Recon** — verify the IP is outside China; check OS, SSH access.
2. **Open questions** — clarify SSH user, cloud firewall layer, device count, budget.
3. **Spec** — produce `docs/spec.md`; you review and approve.
4. **Engineering scaffold** — generate Python code (`subgen` CLI) + tests + deploy scripts.
5. **SSH read-only verification + dry-run** — show you every change before executing.
6. **Deploy + smoke test** — install sing-box, configure firewall, verify handshake.

Each stage waits for your explicit approval. You'll do roughly 6 confirmations end-to-end.

After deploy:

```bash
# Add a device (generates clients/out/<name>.json):
subgen add macbook-alice
subgen add iphone-alice --legacy   # iOS App Store at 1.11.x needs --legacy

# Server picks up new users automatically:
bash infra/push.sh root@<your-host>
```

AirDrop / Signal the per-device `.json` to each user. They import into sing-box client app and connect.

## What you get

- **One sing-box config** that runs both **Reality** and **Hysteria2** on `443/tcp + 443/udp`. The box looks like an ordinary HTTPS endpoint with QUIC.
- **Per-device configs** generated by a small Python CLI (`subgen`). Each device has its own UUID and Hysteria2 password — revoke individually.
- **Cross-platform clients**: sing-box official apps on macOS (SFM), iOS, Android, Linux.
- **Smart routing**: 8 canonical rules (private LAN, domestic DNS, ad-block, China-direct, foreign-via-proxy) + DNS split + Clash API 3-mode toggle.
- **Idempotent deploy script** (`bash` + `nftables` + `chrony` + sing-box apt repo) with `--dry-run`.
- **Provider-agnostic firewall guidance** (GCP / AWS / DigitalOcean / Vultr / bare-metal).
- **Cross-platform smoke test** (macOS BSD + Linux GNU portable).
- **130+ unit tests at ~99.5 % coverage** in the reference scaffold.

## What this skill explicitly will NOT do

- Deploy WireGuard / OpenVPN / IKEv2 for GFW bypass — easily fingerprinted; the skill politely refuses and explains.
- Stand up a public admin panel (wg-easy, 3x-ui) — the panel itself becomes a fingerprint.
- Authenticate to your cloud provider for you — it prints `gcloud` / `aws` commands; you run them.
- Touch your server with anything other than a single read-only SSH inspection until you've reviewed the dry-run.

## FAQ

**Q: I already have a VPS. Can I use it?**
A: Yes, if it's Ubuntu 22.04 / 24.04 / 26.04 or Debian 12 / 13, outside mainland China, with root SSH access. The deploy script is idempotent — it won't clobber existing services unless you've put something on port 443.

**Q: I don't have a domain. Can I skip it?**
A: Yes. Use the server IP directly. Trade-off: if the IP is ever banned by the GFW, you'll need to re-distribute every client config. With a domain, you change one A record and clients recover automatically.

**Q: How long does the first deploy take?**
A: ~1 hour. Most of it is review time at each stage — the actual server-side work is ~5 minutes once you approve.

**Q: How do I add a new device after the initial deploy?**
A: `subgen add <name>` + `bash infra/push.sh root@<host>` + AirDrop the `.json`. About 30 seconds.

**Q: My VPN got banned (IP-level block from China).**
A: Provision a new VPS, point your domain's A record at the new IP, re-run `bash infra/push.sh root@<new-host>`. Clients reconnect automatically within the DNS TTL (default 120 s). If you don't have a domain, you'll re-issue every client config.

**Q: Is this legal?**
A: Self-hosting a VPN is legal in most jurisdictions. Using it to access content that's restricted in your jurisdiction is your responsibility. The skill doesn't run any servers on your behalf and doesn't transit any traffic.

**Q: Why is iOS slower / has weird warnings?**
A: iOS sing-box on the App Store lags upstream by 6–12 months. The skill's `--legacy` flag handles this. See `references/platform-install.md`.

**Q: Can I run multiple servers (e.g., backup region)?**
A: Yes. See `references/multi-deployment.md`. Copy the project tree, change parameters, redeploy. Clients can hold multiple subscriptions and switch in the GUI.

**Q: Where are the secrets stored?**
A: `clients/state.json` (operator's machine, mode 0600). Never committed to git (`.gitignore` excludes it). The skill itself contains no secrets — only methodology and templates.

**Q: I'm not in China — can I still use this?**
A: Yes. The smart routing rules still help: Chinese sites get fast direct routing, foreign sites go through your server. People also use this for accessing geo-restricted content (e.g., a Tokyo server gives you Japanese IP for region-locked services).

**Q: My friend wants this. What do I give them?**
A: Just point them at this repository. They install the skill, they bring their own server, they end up with their own VPN. No shared state between you.

## Hard-won lessons baked in

This skill encodes specific gotchas from real deployments — not theoretical advice. Highlights:

1. **sing-box 1.13 removed multiple legacy schema fields** (string `address` DNS servers, `dns-out` outbound type, inbound `sniff`, `outbound:any` DNS rule, DNS server `detour:direct`). The templates are 1.13+ compliant; tests guard against regressions.
2. **Rule-set URLs with `@`** (e.g. `geosite-apple@cn.srs`) silently 404 in sing-box's HTTP client even when direct curl returns 200. The 4 canonical rule-sets avoid `@` entirely.
3. **`route.default_domain_resolver` is required since 1.12** — without it, sing-box can't resolve outbound `server` domains.
4. **OpenSSL 3.x process substitution** (`<()`) breaks under `sh -c`. The deploy script uses `-pkeyopt ec_paramgen_curve:P-256` instead.
5. **Ubuntu minimal images don't ship rsync.** `push.sh` installs it remotely before the first rsync.
6. **`/etc/sing-box/` must be readable by group `sing-box`** (not `root:root`), or the service won't start.
7. **GCE default service accounts lack `compute` scope** — you can't manage firewall rules from the VM. The skill prints commands for the operator's own workstation.
8. **SFM materializes profiles into `configs/config_<id>.json`**; updating `Profiles/<id>.json` alone is insufficient.
9. **macOS BSD ping uses `-D`** for don't-fragment, Linux GNU uses `-M do`. **BSD mktemp has no `--suffix`.** The smoke test detects the platform and adapts.
10. **iOS sing-box lags App Store** — the `--legacy` flag downgrades the schema for old clients; flip back with `set-legacy --off` when the App Store updates.

See `SKILL.md` for the complete 13-row gotcha table with file references.

## Conventions you should know

- **Tests come before implementation.** The project template Claude generates has `tests/` and `src/vpn_builder/` set up with `pytest-cov --cov-fail-under=80`.
- **Idempotent deploy script.** Safe to re-run; `--dry-run` prints every command + a unified diff of files it would change.
- **No public admin panels.** Add devices via `subgen add` on the operator workstation, then rsync + reload. No web UI.
- **Distribute subscriptions out-of-band.** AirDrop / Signal / encrypted email. **Never** cloud-sync, git, or paste anywhere indexable.
- **Quarterly Reality key rotation, monthly client app update.** Documented in `references/security-runbook.md`.
